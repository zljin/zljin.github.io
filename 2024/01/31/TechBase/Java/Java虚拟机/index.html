堆内存分代机制

对象在内存中的生命周期：
Eden区创建–>经过minorGC存活(复制算法)–>复制到Survivor区–>在Survivor区多次存活(每次存活年龄+1)–>达到晋升年龄阈值(-XX:MaxTenuringThreshold)–>晋升老年代(存放长期存活的对象)–>老年代空间不足–>fullGC(标记整理算法)

优势：

  1. 提高GC效率，针对不同代采用不同的算法和触发频率
  2. 可以控制GC停顿时间，在一个内存密集型应用中通过设置新生代的比例和存活区的比例,减少MinorGC的频率，提高服务响应

类加载与双亲委派机制

类加载机制是JVM将 .class加载到内存，创建对应Class对象的过程，是Java安全模型的基础

类加载的生命周期五个阶段：

  1. 加载：查找字节码文件并创建 Class 对象
  2. 连接：包含验证(确保字节码安全)、准备(为静态变量分配内存并设默认值)、解析(将符号引用替换为直接引用)
  3. 初始化：执行类构造器方法，初始化静态变量
  4. 使用：实例化对象或访问静态成员
  5. 卸载：类被 GC 回收

双亲委派机制

双亲委派的工作流程是：当一个类加载器收到类加载请求时，会先委托给父加载器处理，层层向上传递至启动类加载器。只有当父加载器无法找到该类时(（抛出ClassNotFoundException）)，子加载器才会尝试自己加载

意义:

  1. 安全，防止核心 API 被篡改：例如，即使你自己编写了一个 java.lang.String 类，系统也会优先使用 JDK 内置的 String 类，保证了核心 API 的纯净性和安全性。可以确保类的唯一性：同一个类(由全限定名+类加载器共同确定)在虚拟机中只会被加载一次，避免重复加载。
  2. 性能：避免重复加载类，节省内存
  3. 稳定性：防止类冲突导致的系统崩溃

内存泄漏排查思路

对象不再使用但GC无法回收，导致内存占用持续增长而导致的OutOfMemoryError

常见原因：

1.静态集合对象为强引用，会一直在整个程序的生命周期，按需清理
2.未关闭的资源，如IO流或数据库连接，try-with-resources 即可自动关闭资源
3.ThreadLocal使用不当，记住不用后手动清除
4.监听器和回调未注销
5.内部类和匿名内部类持有外部类引用，如果内部类对象生命周期较长，会导致外部类无法被回收
6.不合理的内部缓存设计

CPU过高排查思路

常见原因：

代码层面：
1.死循环，递归调用
2.频繁GC，内存使用不当
3.线程阻塞，死锁，资源竞争

系统层面：
1.系统负载过高：其他程序占用大量资源
2.内存不足：物理内存使用率过高
3.磁盘 IO 过高：频繁的磁盘读写操作
4.网络 IO 过高：大量的网络请求
5.其他进程影响：系统其他进程占用资源

垃圾回收

Stop-The-World: 指在垃圾回收期间，所有正在运行的程序功能必须暂停，等待垃圾回收完成后才能继续

判断垃圾回收的标准

a. 引用计数算法(无法处理循环引用)
通过判读对象的引用数量来决定是否gc
b. 可达性算法
GCRoot是可达性算法的一个起点，所有线程当前活跃的栈帧都会指向GCRoot,
通过GCRoot找出所有存活的对象，剩下的对象就是被回收的对象

c. 可作为GCRoot的对象
方法区的静态属性和常量引用
虚拟机栈帧引用的对象
本地方法引用的对象
所有线程活跃的对象

d. 不同的引用类型
强引用 通过new出来的对象就是强引用，任何时候都不会gc
软引用 JVM堆空间不足就会gc，有SoftReference持有
弱引用 只要有gc，就会回收
虚引用 随时都可回收

例如，曾优化过一个图像处理应用，通过将大型图像缓存从强引用改为软引用，在内存压力大时自动释放不常用图像，避免了OOM 异常，系统稳定性显著提高。

JIT

Just in time 即时编译技术，JVM监控频繁执行的字节码并编译成本地机器码，大幅提升运行速度
对于复杂计算和大量循环的业务逻辑性能效果明显

-XX:+TieredCompilation 开启分层编译
-XX:CompileThreshold 降低编译阈值

Java应用 kill -9会发生什么

kill -9 强制立即终止进程，不给任何进程处理的机会

1.ShoudownHook会不执行，这个方法通常用于执行资源清理和状态保存等
2.资源泄漏，如连接池，文件流没有自动关闭
3.数据一致性问题，持久化时丢失
4.分布式锁协调，可能导致分布式锁永远不释放

更安全的做法是使用kill不带9参数，这样Java程序可以捕获SIGTERN信号，优雅关闭
