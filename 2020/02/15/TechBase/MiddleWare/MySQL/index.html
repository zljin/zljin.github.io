SQL调优

如何设计表

遵循数据库范式，范式的目的是在设计表时，消除数据冗余，保证数据一致性等，如3NF要求为消除非主属性的传递依赖

基本策略：对于核心业务表严格执行3NF，如
1、供应商信息只存储一次，避免更新不一致；
2、库存变动事务更加可靠；
3、数据迁移时结构清晰

对于高频查询路径适度反范式化，如
1、报表查询需要关联多张表，出现性能瓶颈
2、创建视图

如何索引不失效

1、like查询不以通配符开头
2、join的两表的字段类型要相同
3、最左前缀原则，比如复合索引(a,b,c)但查询只用b,c
4、注意隐式类型转换，比如字符串字段用数字查询
5、索引列不做运算
6、深度分页优化

优化工具 explain

1、查看 type 字段，如果是 ALL 表示全表扫描
2、查看 key 字段，如果是 NULL 表示未使用索引
3、查看 rows 字段，过大需优化
4、Extra， Using index（索引覆盖，不需要回表）、Using filesort/temporary（性能最差）

事务

事务是数据库操作的原子单位，必须满足ACID

通过MVVC原理实现：每行数据都有历史版本，当一个事务执行时，会生成一个快照数据库，包含所有已提交的数据.

事务的隔离级别：为了解决多个事务并发时，产生的下面问题

类型 特点
脏读 读到未提交的数据
不可重复读 同一个事务查询结果不同
幻读 范围查询结果集变化

有四种隔离级别

串行化 完全串行保证强一致性，适合金融场景
可重复读 解决脏读，不可重复读的问题，适合电商场景
读已提交 解决脏读，多数场景表现良好
读未提交 无法解决任何问题
