📁 01. 思维模型（Thinking Models）

核心目的：存储那些能帮你更清晰、更理性地思考世界的底层逻辑框架。

✅ 应该写：

  • 通用思维工具：如 二阶思维（“然后呢？”）、奥卡姆剃刀（如无必要，勿增实体）、逆向思维（“如何确保失败？”）
  • 决策模型：SWOT分析、成本-收益分析、10/10/10法则（这个决定在10分钟/10个月/10年后怎么看？）
  • 认知偏差清单：如 确认偏误、幸存者偏差，并附上你自己的真实案例（比如：“上次选技术栈时，我只看了成功案例，忽略了失败项目”）
  • 系统思维图：简单因果回路图，解释某个现象（如“为什么我总是拖延？”）

❌ 不该写：

  • 抄书定义（除非你用自己的话重述）
  • 和你生活/工作无关的抽象理论

💡 格式建议：每篇笔记 = 模型名称 + 一句话解释 + 我的应用场景 + 行动提示

📁 02. 阅读笔记（Reading Notes）

核心目的：不是记录整本书，而是提取对你有“触动”的观点，并转化为行动。

✅ 应该写：

  • 每本书一个 .md 文件

  • 结构建议：

    • 核心观点（3条以内）
    • 最戳我的一句话（带页码）
    • 和我过去的认知冲突在哪？
    • 我下周要尝试的一个小改变（最关键！）

  • 示例（《被讨厌的勇气》）：

    行动点：当同事质疑我的设计时，先问自己：“这是我的课题，还是他的课题？” 如果是他的情绪问题，我不背锅。

❌ 不该写：

  • 全书摘要
  • 大段摘抄（除非你加了批注）

📌 原则：如果一本书读完后，你写不出一条“行动点”，那它可能不适合你，或者你还没准备好。

📁 03. 学习方法（Learning Strategies）

核心目的：记录经过你验证有效的学习技巧，而不是收藏一堆“看起来有用”的方法。

✅ 应该写：

  • 你亲测有效的技巧：
    • “用 Anki 背 JVM 参数，每天5张卡，两周记住”
    • “看源码前先画 UML 图，效率提升50%”
  • 失败的尝试 + 原因：
    • “尝试早起学习，但效率低 → 因为没睡够，改为晚上深度工作”
  • 工具组合：
    • “Obsidian + Pomodoro + Forest = 我的最佳专注流”

❌ 不该写：

  • 转载的“10个高效学习法”列表（除非你试过）

💡 关键：这里是你个人的“学习操作系统手册”，不是方法论仓库。

📁 04. 灵感碎片（Inbox / Fleeting Notes）

核心目的：临时存放一闪而过的念头、金句、待办事项，避免大脑缓存溢出。

✅ 应该写：

  • 突然想到的问题：“微服务下如何做分布式事务一致性？”
  • 听到的好句子：“完成比完美更重要。”
  • 待查资料：“研究一下 Loom 虚拟线程”
  • 会议中的关键词（来不及整理时）

⚠️ 重要规则：

  • 每周清空一次！把有价值的内容移到其他文件夹，垃圾直接删。
  • 不要在这里长期存放——它只是“中转站”。

📁 05. 优质资源（Curated Resources）

核心目的：建立你的高信噪比信息源库，对抗信息过载。

✅ 应该写：

  • 每个资源一条笔记，包含：
    • 标题 + 链接
    • 一句话价值（“这篇讲透了 Spring 循环依赖的三级缓存”）
    • 适用场景（“适合面试准备” / “适合系统重构参考”）
  • 分类示例：
    • 技术博客（Martin Fowler, InfoQ）
    • 播客（Syntax.fm, The Changelog）
    • 视频课程（推荐的具体章节）

❌ 不该写：

  • 只收藏不评注的链接（那叫“数字囤积症”）

📁 06. 工具清单（Tools & Setup）

核心目的：记录你的生产力工具链配置和使用心得，方便重装系统或分享经验。

✅ 应该写：

  • 开发工具：VS Code 插件清单 + 配置片段
  • 效率工具：Obsidian 插件推荐、Notion 模板
  • AI 提示词（Prompts）：如“帮我用 STAR 法则润色这段项目经历”
  • 系统快捷键：Mac/Linux 常用命令别名

💡 格式：工具名 + 解决什么痛点 + 我的配置/用法

📁 07. 项目复盘（Project Logs）

核心目的：把工作经验转化为可复用的模式，避免重复踩坑。

✅ 应该写：

  • 每个项目一个笔记，包含：

    • 目标 vs 结果
    • 做得好的3点
    • 下次要改进的2点
    • 关键决策背后的思考

  • 示例：

    “上线前没做全链路压测 → 下次所有核心服务必须走 Chaos Engineering 流程”

❌ 不该写：

  • 日常工作流水账

📁 08. 输出作品（Output / Public Writing）

核心目的：存放你对外输出的内容草稿，践行“教是最好的学”。

✅ 应该写：

  • 技术博客草稿
  • 内部分享 PPT 大纲
  • 面试自我介绍模板
  • GitHub README 草案

🌟 终极检验：如果你的知识不能输出，它就还没真正属于你。

✅ 总结：心智花园 ≠ 仓库，而是“生长系统”

表格

05-06 筛选 → 配置
07-08 反思 → 创造

记住：

知识的价值不在于“拥有”，而在于“调用”。
定期回顾、链接、输出，你的“心智花园”才会枝繁叶茂，而不是杂草丛生。

现在，就去创建这些文件夹，从今天的一条“灵感碎片”开始吧！🌱
