Q: Tell me about yourself

Hi, my name is phil.
I graduated from ABC University with a major in Computer Science.
I’ve been working as a software engineer for the five years.
I am good at Java.
I also have a good understanding of DevOps, including Docker, Kubernetes.
I specialize in developing stable APIs for mobile banking systems.
A good example that comes to mind is when I developed CNY transfer and foreign exchange APIs at ABC.
Since release, these APIs have successfully processed many transactions for our application.
That’s all, Thanks.

Q: what are your weakness

I used to spend too much time on details, but now I manage it by setting priorities to deliver on time without compromising quality.

Q: why did you leave your last job

Well, i enjoyed these years that i worked in ABC,  and during that time i’ve learned a lot and i met some amazing people.
To continue growing,I’m now actively seeking new opportunities.

Q: Describe the projects you have worked on.

短回答：

Sure! Recently, I led the migration of our legacy mobile API gateway to a cloud-native architecture.
The goal was to replace old physical gateways, fix a cross-origin logout issue after the frontend moved from Angular to native apps, and connect to our group’s new DSP 3.0 authentication platform.
As the only API developer on the team, I designed a new gateway layer using Spring Cloud Gateway and deployed it on Kubernetes with Istio Service Mesh for secure, observable traffic management. I also set up Kong Gateway as the unified entry point, adding auth and cors plugins.
I worked closely with the frontend team to adjust routing rules and solve CORS issues, and I upgraded downstream services to support SAML 3.0 for DSP 3.0 integration.
We rolled it out gradually with zero downtime, monitored by Splunk and AppDynamics, and the system has been stable ever since.
This project really showed my ability to own a critical component end-to-end while collaborating across teams.

长回答：

Sure! One of the most impactful projects I’ve worked on recently was the cloud-native migration of our legacy mobile API gateway. The main goals were to retire outdated physical gateways, resolve a critical cross-origin logout issue after the frontend migrated from Angular to native apps, and integrate with our group’s next-generation DSP 3.0 authentication platform.
I was the sole API developer on the team, which meant I took full ownership of the gateway layer—from architecture to deployment—and had to proactively coordinate closely with frontend, security, and platform teams.
First, I designed and implemented a new API proxy layer using Spring Cloud Gateway, replacing the old system. It supports dynamic routing, mutual TLS for secure service-to-service communication, and enforces consistent API contracts. To ensure smooth frontend integration, I worked hand-in-hand with the mobile team to redesign routing rules and fix CORS-related logout flows.
Second, I containerized the gateway service and deployed it on Kubernetes, integrating it with Istio Service Mesh to enable fine-grained traffic control, observability, and mTLS across the mesh. At the edge, I configured Kong Gateway as the unified entry point, applying plugins for authentication, rate limiting, and request transformation.
Third, I led the integration with the group’s DSP 3.0 platform, upgrading our downstream services to support SAML 3.0 tokens and ensuring backward compatibility during the transition.
Finally, to minimize risk, I orchestrated a gradual, zero-downtime rollout using canary releases, backed by comprehensive monitoring via Splunk and AppDynamics. The migration was completed successfully with no user impact.
This project really tested and strengthened my ability to drive technical decisions independently while collaborating across functions—especially when bridging backend infrastructure, security requirements, and frontend needs.

why you choose spring boot

We chose Spring Boot for a couple of key reasons. Firstly, It provides a production-ready setup with embedded servers and auto-configuration, which significantly speeds up development. Secondly, It provides an easy way to integrate various dependencies using starters. Finally, It provides an easy way to deploy to production by executing the JAR.
