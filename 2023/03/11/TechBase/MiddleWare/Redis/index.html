Redis

Created2023-03-11|Updated2026-01-07|Middleware
|Post Views:

缓存中间件，也是分布式缓存和分布式锁的核心选项

应用场景

1、查询缓存，注意缓存穿透和雪崩问题
2、短信登录，Redis共享session应用
3、优惠卷秒杀，Redis计数器、Redis分布式锁、Redis三种消息队列
4、排行版功能，基于zset，设置分数
5、附近商户，GeoHash存放地址信息，算距离等

数据类型

数据类型 可以存储的值 操作
STRING 字符串或数值 字符串操作,数字类型的自增自减
LIST 列表 就是队列
SET 无序集合 去重，集合计算，是否存在某元素判断
HASH 包含键值对的无序散列表 存k->Map(k,v) 可存一个bean
ZSET 有序集合 可为元素设置分值，根据分值进行排序，查询分数范围的对象
Geospatial 地理位置 通过Java将经纬度数据导入redis，之后即可通过经纬度计算距离等
Hyperloglog 基数(两个数据集中不重复的元素) 适用于快速合并数据去重，允许一定误差值，效率很高(网站访问量统计)
Bitmap 位存储 二进制记录，适用于统计员工打卡

分布式缓存

为在高并发下减轻数据库压力，提高系统响应速度，需加一层缓存，由于本地缓存承载能力有限，我们会采用多节点分布式缓存

应用场景：
1、页面缓存，如将Web 页面的内容片段,包括HTML、CSS 和图片等缓存 通过ngnix动静分离和云的CDN加速
2、应用缓存，如ORM的二级缓存
3、云计算提供分布式缓存等服务

缓存与数据库不一致问题

1、更新操作数据库后，马上再次更新缓存来实现
2、缓存设置过期时间，等待过期时间后，数据恢复

缓存穿透

用户查询数据，在数据库没有，自然在缓存中也不会，有样就导致用户查询的时候，在缓存中找不到，每次都要去数据库再查询一遍，然后返回空

1、若查询数据结果为空,把空的查询结果的数据缓存起来,下次直接返回

缓存预热

系统上线后，将相关的缓存数据直接加载到缓存系统

缓存降级

在当访问量剧增、服务出现问题是，将非核心业务降级，保证核心服务可用，即使是有损的

缓存过期策略

进行缓存更新核心机制：惰性删除+定时删除

1、定时监控过期缓存，监控内存和过期键统计信息，抽样清空，内存友好

2、惰性删除，访问检查，CPU友好

当有用户请求过来时，再判断这个请求所用到的缓存是否过期，过期的话就去底层系统得到新数据并更新缓存

3、选择合适的策略组合,选择合适的淘汰策略

4、为缓存数据合理设置过期时间，如会话数据30分钟，热点数据5分钟过期 + 延迟双删策略

5、注意避免大量键同时过期导致的”缓存雪崩”

分布式锁

单节点Redis分布式锁

如果业务执行时间超过锁的过期时间，怎么办？

Redisson内部提供了一个监控锁的看门狗，它的作用是在Redisson实例被关闭前，不断的延长锁的有效期。默认情况下，看门狗的检查锁的超时时间是30秒钟，也可以通过修改Config.lockWatchdogTimeout来另行指定。

下面是看门狗机制的关键点：

定时任务 + Lua脚本原子操作 + 递归调度

续期条件：只有在没有指定leaseTime（即使用默认值-1）时，看门狗才会启动。

续期周期：默认每10秒续期一次，将锁的过期时间重置为30秒。

续期操作：通过发送Lua脚本给Redis服务器，重置锁的过期时间。

递归调度：续期成功后，定时任务会递归调用自己，形成持续监控

什么是leaseTime? 即锁的租约时间
