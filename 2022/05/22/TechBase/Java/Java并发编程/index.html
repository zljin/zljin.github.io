多线程编写原则：原子性，可见性，有序性

基础概念

1、进程和线程:进程进行资源调度和分配的的基本单位,由多个线程构成,多个线程共享进程内存,线程是系统最小执行和调度单位
2、并发和并行:并发即在同一时间段内单核CPU对多个任务交替进行，并行即多核同时执行多个任务
3、阻塞和非阻塞反映的是调用者的状态–>调用者是一直等待状态还是可以干其他事情？
4、同步和异步反映的是服务提供者的能力–>服务提供方有结果后能不能通知回调者状态？

创建线程的三种方式

后面统一用CompletableFuture

happens-before 和 as-if-serial 有啥区别和联系

指令重排序,指的是编译器和处理器为了优化程序性能而对指令执行顺序进行重新排列的行为
as-if-serial: 保证单线程程序的执行结果不受指令重排序的影响，允许编译器和处理器进行各种优化
happens-before: 定义了多线程环境下操作间的可见性规则
在多线程的情况下，如果操作 A happens-before 操作 B，那么A的结果对B可见 (不需要时间上的先后，而是逻辑上的先后)

通过volatile的特性建立happens-before关系
volatile: 保证内存可见性，只保证单次写的一致性，适合一写多读的场景. 防重排序

守护线程

后台服务进程，服务主线程，当主线程结束，会立即终止，不会阻止jvm退出.

e.g: GC线程的垃圾回收，监控统计线程等

死锁

多个线程竞争互斥的公有资源而无限等待

1.遇到的场景和如何解决

如线程以不同顺序获取ReentrantLock导致的死锁 —>实现全局锁排序,tryLock设置锁超时
事务A先锁订单再锁库存，事务B相反—>统一约定所有服务必须先锁订单再锁库存 or 将部分同步调用改为异步消息，库存服务引入 Redis 缓存减少数据库锁竞争

threadlocal

线程局部变量：为每一个线程提供一个本地变量(实例副本),专属于这个线程,通过这种方式,避免资源在多线程间共享

ThreadLocal的典型应用
在 Spring 事务管理中，就使用 ThreadLocal 来保存每个线程的连接资源，保证务的隔离性

每次HTTP请求都对应一个线程，线程之间相互隔离
我们项目中用 ThreadLocal 实现了用户上下文传递，避免在方法调用链中层层传递用户信息，ThreadLocal可以在不影响性能的情况下，保存当前线程对应的用户信息从而减少参数传递

使用的注意事项

  1. 任务数很少的时候，不要强行使用，局部变量新建对象即可。
  2. 优先使用框架支持而不是自己随意创建
  3. ThreadLocal的Value是强引用，注意使用后的回收，不然造成内存泄露，无法gc

AQS

AbstractQueuedSynchronizer: JUC基石

  1. 通过volatile state表示资源状态，通过CAS操作避免重量级锁
  2. 通过CLH（CLH 锁是一种基于链表的可扩展、高性能的自旋锁）变体队列管理等待线程
  3. 模版管理，提供了通用并发框架，子类只需实现少量方法
  4. tryAcquire和tryRelease 实现独占模式 如 ReentrantLock，tryAcquireShared和tryReleaseShared 实现共享模式 如 Semaphore
  5. 支持两种策略，fair and unfair

CAS -> compare and swap 乐观锁，适合读多写少，通过添加版本号解决AAB的问题

流程控制

synchronized—->Object:
object.wait();
object.notify();

ReentrantLock:
lock();
unlock();
ReentrantLock–>Condition:
condition.await() //线程阻塞
condition.signal() //线程唤醒

CountDownLatch：倒数计数器，每次执行完一个多线程，计数器减去1,当减为0的时候,主线程开始执行，不能循环利用

CyclicBarrier(班车人到齐了才能开)：循环屏障,允许一组线程相互等待,直到所有线程到达一个公共的屏障点,这些进程再一起执行后续的逻辑

Semaphore(地铁闸机):信号量

  1. Semaphore的构造函数可以传入一个int型整数n，表示某段代码最多只有n个线程可以访问，
  2. 其中acquire(3)可以设置权重，比如我一个线程需要拿3个信号量，如果构造函数只有3，那就别人拿不到，当然释放的时候也要为3
  3. 获取和释放一定要一致，不然不够用
  4. 一般设置公平性要设置为true更为合理，信号量适合执行比较慢的任务

Exchanger:
线程协作工具类，用于两个线程之间的数据交换，两个线程都执行到exchange方法，那么这两个线程就可以交换数据

并发容器

ConcurrentHashMap（线程安全的Map）
CopyOnWriteArrayList 先拷贝一份数组,在拷贝的数组中写数据,写完后丢弃原来的数组,指向拷贝后的数组。适合读多写少的场景，保证最终一致性。
AtomicInteger (原子类) 实现原子累加，基于CAS

BlockingQueue，队列满或者为空，线程就被阻塞，可控制线程之间的通信

foobar 六脉神剑之一

锁

乐观锁与悲观锁

悲观锁也叫互斥同步锁，每次访问都加锁，适合写多读少
应用场景：select for update,synchronized

乐观锁也叫非互斥同步锁,适合读多的场景
应用场景：AQS(CAS),AtomicInteger

可重入锁

先第一次申请这把锁后，当再次申请这把锁时，无需提前释放这把锁，而是可以直接继续使用这把锁.(可重入)
同一个线程可以多次获取同一把锁,而不需要强制先释放
好处：避免死锁，减少频繁解锁的流程

共享锁(读锁)和排他锁(写锁)

读写锁是一把锁，有两种锁定关系而已，如把获取写锁理解为–>把读写锁进行写锁定

自旋锁和阻塞锁

自旋锁就是当前面线程的锁定时，我们自己先自旋一下（先占着CPU资源不放，如果太久则太耗CPU资源），前面线程释放后，则进行加锁。这样可以避免切换线程的开销。

锁升级和降级

synchronized 的锁是不能降级的。锁升级过程（从偏向锁到轻量级锁再到重量级锁）是单向的。

实际编程中如果需要类似锁降级的功能，可以考虑ReentrantReadWriteLock
