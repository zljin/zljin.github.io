数据库范式

范式的目的是在设计表时，消除数据冗余，保证数据一致性等，如3NF要求为消除非主属性的传递依赖

基本策略：

对于核心业务表严格执行3NF，如
1)供应商信息只存储一次，避免更新不一致；
2)库存变动事务更加可靠；
3)数据迁移时结构清晰

对于高频查询路径适度反范式化，如

  1. 报表查询需要关联多张表，出现性能瓶颈

查询语句的执行顺序

理解执行逻辑顺序能有助我们编写高性能SQL

1、首先处理 FROM 和 JOIN 确定数据来源，这是查询的起点
2、然后应用 WHERE 条件尽早过滤数据，性能优化的关键点
3、接着执行 GROUP BY 分组，这通常需要排序或哈希操作，需要注意 WHERE 与HAVING 的区别，WHERE 在分组前过滤更高效
4、之后处理 SELECT 选择列，包括计算表达式
5、最后应用 ORDER BY 和 LIMIT

索引

索引失效：本质是破坏了B+树的有序查找机制

最常见的失效场景和优化方向

1、违反最左前缀原则，比如复合索引(a,b,c)但查询只用b,c 列

2、是对索引列做运算或函数，如 YEAR(create_time)=2023

3、隐式类型转换，比如字符串字段用数字查询
4、范围查询阻断后续列，如 age>20 AND name=’张三’只能用到 age 列索引
5、使用!= 或者 <> 或者IS NULL
6、like以通配符开头
7、使用OR时，两边的字段都要有索引
8、进行 join 的字段两表的字段类型要相同，不然索引实效
9、深度分页优化

诊断方法：explain

1、查看 type 字段，如果是 ALL 表示全表扫描
2、查看 key 字段，如果是 NULL 表示未使用索引
3、查看 rows 字段，过大需优化
4、Extra， Using index（使用了覆盖索引）、Using filesort/temporary（性能最差，要避免）

什么是覆盖索引

索引包含了查询所需的所有字段，查询可以直接通过索引获取数据而无需回表

适合查询少量列，但数据量大的表

减少索引维护成本，索引越多，写操作成本越高

唯一索引和主键索引

主键索引作为聚簇索引，决定了数据的物理存储顺序，叶节点直接包含完整行数据，且不允许 NULL

唯一索引是二级索引，叶节点只存储主键值，需要回表查询，且允许多个NULL值
