多线程

1.并发与并行

  • 并发:指两个或多个事物在同一个时间段内发生(并发是轮流处理多个任务)
  • 并行:指两个或多个事物在同一个时刻发生(并行是同时处理多个任务)

2.线程与进程

进程

  • 进程是程序的一次执行过程，是资源分配的最小单位,是抢占处理机的调度单位.
  • 独占内存空间,保存各自运行状态,相互间不干扰且可以互相切换,为并发处理任务提供可能.

线程

  • 线程是进程中的一个执行单元,负责当前进程中程序的执行,一个进程中至少有一个线程.
  • 共享进程的内存资源,相互间切换更快速,支持更细粒度的任务控制,使进程内的子任务得以并发执行

线程的状态

3.线程的两种创建方式

  • 新建一个类 继承Thread类,重写run()方法
  • 新建一个类 实现Runnable接口重写run()方法

4.守护线程 setDaemon(boolean)

  • 设置一个线程为守护线程,该线程不会单独执行,当其他非守护线程都执行结束后,自动退出[守护线程也可以守护main线程]

5.线程的同步问题

多个线程抢占一个资源,保证每次只有一个线程抢占一个资源,其他必须等待这个线程结束才能继续公平的抢(火车站抢票)

synchronized锁

基本特性

互斥锁的特性:互斥性(保证同一个时间只有一个线程持有对象锁),可见性(一个线程对共享变量修改后其他线程可见)

synchronized底层实现原理

java对象头,Monitor:每一个Java对天生自带了一把看不见的锁

Lock锁(互斥锁- 条件变量)

  • Lock锁也称同步锁，方法如下：
  • public void lock() :加同步锁。
  • public void unlock() :释放同步锁。
  • ReentrantLock将锁对象化

6.线程间的通信,同步锁

  • 什么是线程间的通信
    • 多个线程在处理同一个资源，但是处理的动作（线程的任务）却不相同(生产者消费者问题)
  • 锁对象调用wait() ,notify() 和 notifyAll()使用要求:
    • wait()和notify()必须在同一锁对象使用
    • wait()和notify()是Object的对象方法
    • 此方法必须在同步方法或者同步方法块中实现

面试常考

  • sleep和wait的区别?
    • Thread.sleep Object.wait
    • sleep()方法可以在任何地方使用,wait()方法只能在synchronized方法或块中使用
    • 主要本质区别:Thread.sleep只会让出CPU,不会导致锁改变,Object.wait不仅会让出CPU,还是释放已占有的同步资源锁

  • Thread类中的start()和run()方法有什么区别?

    • 调用start()方法会创建一个新的子线程并启动,run()方法只是Thread的一个普通方法调用(主线程调用)
    • Thread类继承Runnable接口

  • Thread.join()方法?

    • join()方法主要功能是实现同步,使线程从并行执行到串行执行,join()里面添加参数则在这个参数时间后恢复并行

  • 如何给run方法传参?

    • 构造方法,成员变量方法,回调函数

  • 实现线程返回值的方法?

    • 主线程等待法
    • 使用Thread类的join()阻塞当前线程以等待子线程处理完毕 ? 精度不够
    • Callable接口实现:通过FutureTask 或者 线程池获取

  • notify和notifyall的区别?

    • 锁池: 线程A拥有某个对象的锁,线程B,C想调用此对象的某个synchronized方法,但A占用了该锁,所以B,C要在锁池等待
    • 等待池:线程A调用wait()方法则此线程会释放对象锁然后进入到该对象的等待池中,进入到等待池的线程不会竞争该对象的锁
    • 作用:唤醒lock.wait()
    • notifyAll 会让所有处于等待池的线程全部进入锁池去竞争获取锁的机会
    • notify 只会随机选取一个处于等待池中的线程进入锁池去竞争获取锁的机会

  • yield?

    • Thread.yield(),会给线程调度器一个当前线程愿意让出CPU使用的暗示,一般会忽略

  • 如何中断线程?

    • interrupt(),通知线程应该被中断了(暗示),一般会忽略

类的加载机制

类加载器机制(类加载只执行一次)

类是在运行期间第一次使用动态加载,如果一次性加载会占用很多内存.

类加载过程

加载->验证->准备->解析->初始化

加载:完成3件事(此过程通过classloader完成):
  • 通过类全名获取该类二进制字节流
  • 将字节流的静态存储结构转换到方法区的运行存储结构
  • 在堆中生成代表此类的Class镜像作为方法区对这些数据的访问入口
验证:确保次类字节流文件信息符合JVM的要求
准备:为类变量分配内存并初始化
解析:虚拟机常量池的符号引用替换为直接引用

什么是符号引用和直接引用???

初始化:

‘<’clinit’>’:编译后在字节码文件中生成的clinit方法,称为类构造器(将静态语句块,静态变量初始化收敛到此方法中)

‘<’init’>’:编译后在字节码文件中生成的init方法,称为实例构造器(将语句块,变量初始化收敛)

  • 初始化是JVM执行类构造器’<’clinit’>’方法的过程.在准备阶段类变量已经赋过一次系统的初始值,在初始化阶段可以主观计划初始化类变量和其他资源.
  • JVM会保证一个类执行’<’clinit’>’方法在多线程环境下正确的加锁和同步.
类的初始化时机:

主动引用(初始化过程会被触发执行)

  • 遇到new字节码指令
  • 反射调用
  • 初始化一个子类,若父类还没初始化
  • JVM启动,用户指定执行的主类(含有main方法的类),先初始化此类

被动引用(除主动引用外,所有引用类的方式都不会触发初始化)

  • 子类调用父类静态字段
  • 直接引用常量(常量在编译阶段会存入调用类的常量池中,本质上并没有直接引用到定义常量的类,因此不会触发定义常量的类的初始化)
  • 通过数组定义来引用类,不会触发此类的初始化。该过程会对数组类进行初始化,数组类是一个由虚拟机自动生成的、直接继承自 Object 的子类,其中包含了数组的属性和方法

类与类加载器

什么是类加载器(classloader)?
java核心组件,所有class必须由它加载classloader负责将Class文件里
二进制数据流装载进系统,然后交给jvm进行连接,初始化操作

类相等判断标准?
类本身相同且在同一个类加载器中进行加载,这是因为每一个类加载器都拥有一个独立的类名称空间.

classloader分类
  • Bootstrap ClassLoader,启动类加载器 $JAVA_HOME/lib (C++编写,加载核心库java.* )
  • External ClassLoader,外部类加载器 $JAVA_HOME/jre/lib/exts (Java编写,加载扩展库javax.*)
  • Application ClassLoader,应用程序类加载器 $CLASSPATH(Java编写,加载程序所造目录)
双亲委托机制
  • 自顶向下尝试加载类,自底向上检查类是否已经加载

作用:加载位于rt.jar包中的类java.lang.Object,不管是哪个加载器加载这个类,最终都是委托给顶层的启动类加载器进行加载,
这样就保证了使用不同的类加载器最终得到的都是同样一个Object对象.(可以避免多份同样的字节码加载)

synchronized+DCL(Double check lock)

基于类的初始化

利用classloder的机制来保证初始化instance时只有一个线程。
JVM在类初始化阶段会获取一个锁，这个锁可以同步多个线程对同一个类的初始化。

对于每一个类或者接口C,都有一个唯一的初始化锁LC与之相对应。从C到LC的映射，由JVM的具体实现去自由实现。
JVM在类初始化阶段期间会获取这个初始化锁，并且每一个线程至少获取一次锁来确保这个类已经被初始化过了

JVM中的GC回收机制

字数统计: 1.7k 字 | 阅读时长 ≈ 6 分钟

垃圾收集主要针对堆和方法区这种线程共享的区域,因为像栈和PC区域属于线程私有,当线程的生命周期结束时也就会消失

判断对象是否是个垃圾

1.引用计数算法(无法处理循环引用)

2.可达性算法

  • 通过判断对象的引用链是否可达来决定对象是否可以回收

  • GCRoots包含以下内容
    • 虚拟机栈的引用的对象
    • 方法区中常量的引用的对象
    • 方法区中类静态属性所引用的对象
    • 本地方法栈中JNI(Native方法)的引用对象
    • 活跃线程的引用对象

垃圾回收算法

1.标记清除算法(Mark and Sweep)

  • 标记:从根集合进行扫描,将存活对象标记
  • 清除:从堆内存从头到尾线性遍历,回收不可达对象
  • 缺点碎片化,不能压缩存活对象,在CMS中采用此算法后还要进行一次gc

2.复制算法(Copying)

  • 分为对象面和空闲面,对象创建在对象面,存活的对象存放在空闲面,将对象面中的所有对象从内存清除
  • 优点:解决碎片化,顺序分配简单高效,适用于对象存活率低的情况(适用于年轻代)

3.标记整理算法(Compacting)

  • 标记:从根集合进行扫描,将存活对象标记
  • 清除:移动所有存活对象,按照内存依次排序,然后将末端内存地址以后的内存全部回收
  • 优点:1.避免内存的不连续性 2.不用设置两块内存互换 3.适用于对象存活率高的情况(适用于老年代)

4.分代收集算法(Generational Collector)

  • 按照对象生命周期的不同划分区域进行不同的垃圾回收算法

GC的分类

年轻代:尽可能快速地收集掉那些生命周期短的对象
由Eden区 和两个Survivor区(From区和To区,此两区顺序不是绝对)

回收演示过程

  • 对象如何晋升到老年代
    • 经历一定Minor次数依然存活的对象(-XX:MaxTenuringThreshold)
    • Survivor区中存放不下的对象
    • 新生代的大对象(-XX:+PretenuerSizeThreshold)
    • -XX:SurvivorRatio:Eden和Survivor的比值
    • -XX:NewRatio: 老年代和年轻代值的空间比值

老年代:存放生命周期较长的对象

  • Minor GC:回收新生代,因为新生代对象存活时间很短,会频繁执行,执行的速度快

    触发条件:当Eden空间满时,触发

  • Full GC:回收老年代和新生代,老年代对象其存活时间长,Full GC很少执行,且速度慢

    触发条件:
    1.老年代空间不足 2.永久代空间不足 3.调用System.gc() 4.Minor GC晋升老年代的平均大小大于剩余空间

垃圾收集器

两个概念

  • Stop-the-World:JVM执行GC要停止应用程序的执行,任何GC都会发生,通过优化stw的时间来提高程序性能
  • Safepoint:分析过程中对象引用关系不会发生变化的点(一般出现在:方法调用,循环跳转,异常跳转)
  • JVM的运行模式:分为Client运行模式和Server运行模式

垃圾收集器的关系图

  • 单线程与多线程：垃圾收集器使用线程数.
  • 串行与并行：串行指垃圾收集器与用户程序交替执行(需要停顿用户程序),并行是同时执行(不需要停顿用户程序),除了CMS/G1都是串行。

年轻代常见垃圾收集器

Serial收集器(-XX:+UseSerialGC,复制算法,单线程,简单高效,JVM的Client模式下默认的年轻代收集器)

ParNew收集器(-XX:+UseParNewGC,复制算法,多线程,和前面类似,多核执行才比前面有优势)

Parallel Scavenge收集器(-XX:+UserParallelGC,复制算法,多线程,吞吐量优先收集器,多核才有优势,JVM的Server模式下默认的年轻代收集器)

  • 吞吐量:运行用户代码时间/(运行用户代码时间+垃圾收集时间)
  • 其它收集器目标是尽可能缩短垃圾收集时用户线程的停顿时间(用户交互时间越短能提高用户体验)，而它的目标是达到一个可控制的吞吐量(可高效率利用CPU时间,快速完成程序运算任务)

老年代常见的垃圾收集器

Serial Old收集器(-XX:+UseSerialOldGC,标记整理算法,单线程,Client模式下默认老年代收集器)

Parallel Old收集器(-XX:+UseParallelOldGC,标记整理算法,多线程,吞吐量优先)

CMS收集器(-XX:+UseConcMarkSweepGC,标记清除算法)

  • 流程:
    1. 初始标记:stop-the-World
    2. 并发标记:并发追溯标记,程序不会停顿
    3. 并发预处理:查找执行并发标记阶段从年轻代晋升到老年代的对象
    4. 重新标记:暂停虚拟机,扫描CMS堆中的剩余对象 (需要端正stw)
    5. 并发清理:清理垃圾对象.程序不会停顿
    6. 并发重置:重置CMS收集器的数据结构

  • 特点:并发标记的过程就是和用户同时工作(边丢垃圾边打扫)

G1收集器(-XX:+UseG1GC,复制+标记-整理算法)

  • 特点:
    1. 并行并发,分代收集,空间整合,可预测停顿
    2. 将整个Java堆内存划分成多个大小相等的Region,年轻代和老年代不再物理隔离
